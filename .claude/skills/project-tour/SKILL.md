# 生成项目 AI 导览(AI_GUIDE.md)

目标:为一个项目产出一份**密集、自足**的 `AI_GUIDE.md` 放在项目根,让"只读问答"(尤其飞书机器人的只读查询)能**不打开其他文件就答对 80% 的常见技术问题**,只有深挖时才按它的文档索引去读 `docs/`。**一次生成、多次省 token**(飞书查询侧只在建会话时注入一次,之后 `--resume` 走 prompt cache)。

## 流程

1. **确定项目根** `<PROJ>`:用户给的绝对路径;没给就用当前目录。先 `ls` 看清顶层结构。

2. **生成符号级 repo map**(抽签名、去实现体,排除数据/产物,避免逐个读大文件):
   ```
   npx --yes repomix --compress --style markdown \
     --ignore "test_data/**,**/*.mat,**/RawData/**,**/__pycache__/**,*.png,*.jpg,*.csv,dist/**,build/**,node_modules/**,.git/**" \
     -o "<PROJ>/.repomix.md"
   ```
   - `--compress` 用 Tree-sitter 只留函数/类签名,省 ~70% token。
   - 没有 Node/repomix 时,退化方案:用 Glob 列源码文件树 + Read 各入口脚本和每个模块的 `def`/`class` 头部。
   - **成千上万的数据文件(.mat/.png/.csv 等)必须 ignore,绝不逐个读。**

3. **理解项目**:读 `<PROJ>` 下的 `docs/`、`README*`,以及第 2 步的 `.repomix.md`。已有的中文文档优先复用(别重写,导览只做索引 + 高频答案)。

4. **写 `<PROJ>/AI_GUIDE.md`**:
   - **第一行必须是生成标记**(飞书只读查询靠它检测导览是否过时):先取当前 git 短 hash `git -C "<PROJ>" rev-parse --short HEAD`(不是 git 仓库就写 `nogit`),文件第一行写成 `<!-- project-tour · generated <本地日期 时:分> · git <hash> -->`。
   - 其后严格按 `references/template.md` 的 8 节:① 一句话定位 → ② 架构与数据流(ASCII 图即可)→ ③ 模块职责表(路径 → 一句职责 → 关键函数)→ ④ 测试/运行流程(入口、命令、参数、依赖、跑一次的步骤)→ ⑤ **数据格式与命名约定(逐字段解码一个真实文件名样例)** → ⑥ FAQ(同事最可能问的 5–10 个问题,直接给答案)→ ⑦ 术语表(中英对照)→ ⑧ 文档索引(`[docs/x.md](相对路径):说明`)。

5. **控制规模**:200–400 行 / 约 2–4K token,自足;细节留 `docs/`,导览只做"索引 + 高频答案"。

6. **用中文写**。删除临时 `.repomix.md`(或加进 `.gitignore`)。

7. **收尾提示用户**:代码结构 / 数据格式大改后重跑本 skill 刷新;若该项目已被飞书机器人查询过,提示在飞书点该项目卡的「🧹 清空查询记忆」或发「忘记查询」,让新导览生效。

## 约束
- **只读分析项目代码**,不改任何源码;只新增/覆盖 `<PROJ>/AI_GUIDE.md`(和临时 `.repomix.md`)。
- 数据目录必须 ignore,绝不遍历读取。
- 面向"被 AI 一次读懂并回答问题"来写——**密度优先**,不是给人看的教程。
